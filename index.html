A programming environment should be perceived as an environment which supports composing systems and setting up their reaction patterns whichever means crystalize such an environment.
Dynamake is an attempt to envision, design, and implement an interactive code extrinsic programming environment. This environment is to require only little configuration effort before getting the hands dirty. In contrast to most programming environments, Dynamake is supposed to be a self-sufficient programming environment integrable with the "outer world". There are many sources of inspiration driving this project. Dynamake is centered around the concept of models and tools to interact with these models. It is hoped that tools at some point, themselves, are represented using models. A general design strategy is to attempt to merge the concept of program and process in strict sense - rather adopting the program/process dichotomy.
A model should be perceived as a sensitive entity observing its surroundings constantly where each observation may provoke a reaction. One of the more tacit examples of this, is deriving models from other models.
So much experimentation has been performed on software systems and their potential crystalizations. Especially the 60's and 70's seem like golden decades of such work. One can't avoid being inspired from the great work done in this era. One of the major examples here is the mother of all GUIs: Sketchpad by Ivan Sutherland in 1963 - an incomprehensible leap within computer graphics and user interfaces.
Using Dynamake across a network is not particularly well-defined yet. However, a vague driver of this aspect is context sensitivity and implicit recognition of Dynamake instances in the surroundings - whatever "surroundings" (and "Dynamake instances" for that matter) implies. Further, no configuration of connections to central servers should be necessary.
Uniting program and process, as concepts, into a single concept is definitely an old idea which has been incarnated in many forms to many different degrees. In relation to software, one of many examples are the original perceptions of object oriented programming where objects were thought of as "...being like biological cells and/or individual computers on a network, only able to communicate with messages".
This dichotomy is concerned with separating programs from processes, where processes are seen as running programs. This separation has implicated a couple of less-than-good effects.
  • One of the bigger issues with this dichotomy is that it has been propogated through many layers of software and hardware. Especially the popular OS's have taken a big hit on this with heavy weight, and non-scalable, processes where the end-users have become hostages of this phenomena. Programs are basically just seen as a special kind of file where files are completely passive elements in themselves. Processes are usually transient in nature and any long-term persistence parts are stored in files some way or another and then reloaded each time the processes are started. A special case of processes are services (still, usually, transient) which are loaded each time an OS is booted. Instead, an OS could just always be running; rather than booting one would just simply continue. This is by no means a new idea and attempts of such an OS has been made on multiple occasions. Of course, like the best user interface, the best OS is none.
  • Another issue is that separating programs from their processes has provoked databases to become necessary evils. Before one even decides that long-term persistence is necessary for a software system, a need to (continously and awkwardly) integrate to an external database system of one form or another becomes the first step of such a practically essential property of a system. Such an integration really shouldn't be necessary - at least at the beginning of the development of a software system.
  • Hardware has been affected by having the concept of memory and the concept of processor incarnated with only little overlap; memory does little processing and processers have little memory. An overlap is essential among these concepts for doing computation and it could be interesting to see computers only consisting of incarnations of a concept merged more or less equally of these two concepts.
A model can be derived from another model. A model, from which other models are derived from, is called a master. Initially, a derived model seems like a traditional copy. However, as changes are made to its master, these changes are implicitly reflected in the derived model without affecting the local history in the derived model. Basically, this kind of relation works by the masters forwarding changes in their local history to their derivations, which then observe that there are new changes to apply and then update according to the newly observed forwarded changes.
When network support is implemented this feature is supposed to be streamless; meaning no difference should exist whether changes are propogated within the same OS process, the same machine, or the same surroundings.
A Java Runtime Environment is required to start playing with Dynamake. Alas, currently, there is no ready-built release available, but such a release is planned to be made and published on GitHub autumn 2014. However, Eclipse can be used to build such a release relatively easy getting the sources on GitHub using Git.
There is no need for setting up a database of any sort to start persisting your software and their changes because Dynamake implicitly does this for you.

What is Dynamake not about?
There is no aspiration of developing a programming environment for solving every kind of problem. Instead, the environment is developed in an open-ended, experimental, and intuitive manner with a focus on a platform for making software.
Why was this project started?
There were/are many reasons motivating this project. Lets look at a couple of'em.
Most popular programming environments seem to dichotomize programs and processes simply by centering around code. What if one attempts to not adopt this dichotomy?
Most popular programming environments are oriented around some sort of text-based, or similarly structered, programming language. What if no language is available at all - how does one program?
What is the current state of Dynamake?
Dynamake is a rather new project, started April 2014, but some of the envisioned attributes are starting to emerge. There is little configuration required to start trying out Dynamake.
What about the future of Dynamake?
One of the major works in the horizon is support for using Dynamake across a network.
